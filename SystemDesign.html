Product Overview
What you're building — and why it will work
The Vision

A beautiful, minimal desktop app that replaces the chaos of doom-scrolling with a curated, AI-filtered news experience. Users define their interests once — broadly or with surgical precision — and the app delivers clean, distraction-free briefings on their schedule. No ads. No clickbait. Just signal.

Core Problem Solved
  • News is noisy, clickbait-heavy, and time-consuming
  • No tool lets you define exactly what you care about
  • Desktop = focus-first reading environment
  • TL;DR-first format respects the user's time
  • AI dialogue for topic setup beats any dropdown
Key Differentiators
  • LLM-powered topic refinement via conversation
  • AI-generated TL;DR (3–4 bullets) for every article
  • ElevenLabs audio playback — listen while you work
  • Frequency control: daily digest / weekly / realtime
  • Zero clutter: no ads, no tracking, no filler

News Pipeline Data Flow
// Step 1 — Trigger Cron Job fires per user schedule (daily 8am / weekly Mon) → Reads user topic configs from Supabase DB → Enqueues fetch jobs in BullMQ (Upstash Redis) // Step 2 — Fetch Worker picks up job → Calls Perplexity sonar-pro with engineered query → Returns 5–10 articles with source URLs + raw text // Step 3 — Summarize GPT-4o-mini via Vercel AI SDK generateObject() → Input: raw article text → Output: { tldr[], body, readMinutes, sentiment } → Dedup via SHA256 URL hash → store in Supabase // Step 4 — Serve Electron app pulls via Supabase JS client (JWT auth) → Renders home dashboard with topic cards + TL;DRs → Audio generated lazily on first play button click
Electron ↔ Backend Security Model

The Electron app is a read-only consumer. It holds only a Supabase user JWT (stored via Electron's safeStorage API, encrypted at rest). All writes and sensitive operations go through Supabase Row Level Security or Edge Functions with a server-side service key. No API keys for Perplexity, OpenAI, or ElevenLabs ever touch the Electron client.

Realtime Updates

When a new article batch is processed, the worker marks articles as ready. Supabase Realtime (Postgres CDC) pushes a notification to any connected desktop clients for that user. The app can display a subtle "N new articles" badge without requiring a reload — works like a live feed.

AI & Agent Design
Three purpose-built agents — topic refinement, news search, summarization
Agent 1 — Topic Refinement (Onboarding Dialogue)

User enters a free-text sub-topic. The LLM runs a max 3-round dialogue to clarify intent, then produces a structured topic config object stored in the DB.

User
"TSMC company finances"
NewsFlow Agent
"Got it — a few quick questions to sharpen your feed:
① Track earnings calls & quarterly financial results?
② Include deal announcements (fab expansions, partnerships)?
③ Stock price movements & analyst ratings?"
User
"Yes to ① and ③, skip ②"
Stored Config
{ topic: 'TSMC', signals: ['earnings', 'analyst_ratings', 'stock_moves'],
  exclude: ['deals'], category: 'Finance' }

Implemented with streamText() from Vercel AI SDK. System prompt ends with instruction: "After final confirmation, output ONLY a JSON config object and nothing else." Parse and store in Supabase topics table.

Agent 2 — News Fetcher (Perplexity sonar-pro)
  • Worker constructs targeted query from topic config: "TSMC earnings financial results analyst ratings site:reuters.com OR site:ft.com OR site:bloomberg.com"
  • Perplexity returns 5–10 articles with source URLs, publication dates, and raw content — no scraping needed
  • Deduplication: SHA256 hash of source URL stored per topic. Articles already seen are skipped.
  • Runs on user's schedule via BullMQ job queue — not on every app launch
  • Batch optimization: users with identical broad topics (e.g. "AI news") share one Perplexity call → split results per user
Agent 3 — Article Summarizer (GPT-4o-mini + Zod)

Uses Vercel AI SDK's generateObject() with a Zod schema — guarantees structured output every time, no JSON parsing errors.

// Zod schema for article output const ArticleSchema = z.object({ tldr: z.array(z.string()).min(3).max(4), // 3–4 bullet TL;DR body: z.string(), // cleaned ~300 word body readMinutes: z.number(), // estimated read time sentiment: z.enum(['positive', 'negative', 'neutral']), keyEntities: z.array(z.string()), // TSMC, Jensen Huang... }) // System prompt key instructions: // "Remove all ads, promotional content, author bios, and boilerplate." // "Body must be under 350 words. Target 2-minute read." // "TL;DR bullets: one key fact per bullet, no fluff."
ElevenLabs Audio — Lazy Generation Strategy

Audio is never pre-generated. It's created on first play and cached permanently. This keeps costs near zero for articles users never listen to.

Cost: ~$0.36/article (2k chars × $0.18/1k). Most users listen to <10 articles/month → ~$3.60 worst case. Usually far less. Delete audio files for articles older than 90 days to manage Storage costs.

Row Level Security (RLS)

Every table has RLS enabled. Users can only read and write their own rows — enforced at the database level, not in application code. Articles are accessible only through topics owned by the user. Audio generation and article writes happen via Supabase Edge Functions using a server-side service key — never exposing elevated privileges to the Electron client.

Cost Optimization Strategies
  • ElevenLabs: Lazy generation only — never pre-generate. Delete cached audio files after 90 days. This alone saves ~60% of audio cost.
  • Perplexity: Batch users with identical broad categories (e.g. all "AI news" subscribers) into a single search call, then split results.
  • GPT-4o-mini not GPT-4o for summarization — 25× cheaper, same quality for structured extraction tasks. Reserve GPT-4o for the onboarding dialogue only.
  • Supabase Storage lifecycle: Auto-delete audio files and raw article cache older than 90 days via storage policies.
  • Deduplication: URL hash prevents re-summarizing the same article even if it appears in multiple topic searches.
